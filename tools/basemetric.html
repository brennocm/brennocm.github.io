Guia Definitivo: Como Classificar Vulnerabilidades

1. O que é CVSS?

O Common Vulnerability Scoring System (CVSS) é um framework aberto e padronizado para avaliar a severidade de vulnerabilidades de segurança em sistemas. Desenvolvido pelo FIRST (Forum of Incident Response and Security Teams), o CVSS fornece uma pontuação numérica de 0.0 a 10.0 que reflete a gravidade de uma vulnerabilidade.

Por que usar CVSS?

  • Padronização: Linguagem comum entre equipes de segurança, desenvolvimento e gestão
  • Priorização: Ajuda a decidir quais vulnerabilidades corrigir primeiro
  • Comunicação: Facilita reportar riscos para stakeholders não-técnicos
  • Compliance: Requisito em normas como PCI-DSS, ISO 27001 e frameworks de GRC

Tabela de Severidade

Score Severidade Ação Recomendada
0.0 Nenhum Informativo apenas
0.1 - 3.9 Baixo Correção em ciclo normal
4.0 - 6.9 Médio Correção prioritária
7.0 - 8.9 Alto Correção urgente
9.0 - 10.0 Crítico Correção imediata

2. Entendendo as Métricas

Attack Vector (AV) - Vetor de Ataque

Define como o atacante pode alcançar o componente vulnerável:

Network (N) Score: Alto

Explorável remotamente via Internet. Exemplo: API exposta na web, servidor público.

Adjacent (A) Score: Médio-Alto

Requer acesso à mesma rede (Wi-Fi, LAN, Bluetooth). Exemplo: ARP Spoofing em rede corporativa.

Local (L) Score: Médio

Requer acesso local ao sistema. Exemplo: Escalação de privilégio após login SSH.

Physical (P) Score: Baixo

Requer acesso físico ao dispositivo. Exemplo: Cold boot attack, USB malicioso.

Attack Complexity (AC) - Complexidade do Ataque

Condições fora do controle do atacante que precisam existir:

Low (L) Score: Alto

Ataque repetível, sem condições especiais. Exemplo: XSS refletido simples.

High (H) Score: Baixo

Requer condições específicas (timing, config especial). Exemplo: Race condition.

Privileges Required (PR) - Privilégios Necessários

Nível de autenticação/autorização antes do ataque:

None (N) Score: Alto

Sem autenticação. Exemplo: SQL Injection em página de login.

Low (L) Score: Médio

Usuário comum autenticado. Exemplo: IDOR acessando dados de outros usuários.

High (H) Score: Baixo

Privilégios administrativos. Exemplo: RCE via painel admin.

User Interaction (UI) - Interação do Usuário

O ataque requer ação de outro usuário?

None (N) Score: Alto

Totalmente automatizado. Exemplo: Worm, exploit remoto.

Required (R) Score: Médio

Vítima precisa clicar/abrir algo. Exemplo: Phishing, XSS persistente + clique.

Scope (S) - Escopo

A vulnerabilidade afeta outros componentes além do vulnerável?

Unchanged (U)

Impacto limitado ao componente. Exemplo: DoS em um serviço específico.

Changed (C)

Afeta outros sistemas/componentes. Exemplo: VM escape, XSS que rouba cookies de outro domínio.

Impactos (CIA Triad)

Avalie cada dimensão da tríade CIA:

  • Confidentiality (C): Acesso não autorizado a informações
  • Integrity (I): Modificação não autorizada de dados
  • Availability (A): Interrupção do serviço ou sistema

Para cada um: None (N) = sem impacto | Low (L) = impacto parcial | High (H) = impacto total

3. Exemplos Práticos

Log4Shell (CVE-2021-44228) Score: 10.0 CRÍTICO
CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:H

Análise: RCE remoto via Internet, sem autenticação, sem interação do usuário, com impacto total em todos os pilares CIA e afetando sistemas além do vulnerável.

IDOR com Vazamento de Dados Sensíveis Score: 7.5 ALTO
CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N

Análise: Acesso remoto a dados sensíveis sem autenticação, mas apenas impacto em confidencialidade.

XSS Stored com Interação Score: 6.1 MÉDIO
CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S:C/C:L/I:L/A:N

Análise: Requer que vítima acesse a página infectada, mas permite roubo de sessão.

Information Disclosure via Error Message Score: 3.7 BAIXO
CVSS:3.1/AV:N/AC:H/PR:N/UI:N/S:U/C:L/I:N/A:N

Análise: Vazamento menor de informações que requer condições específicas.

4. Boas Práticas

+
Seja consistente

Use os mesmos critérios em toda a organização. Crie um guia interno de classificação.

+
Considere o contexto

Use métricas Environmental para ajustar ao seu ambiente (criticidade do ativo, controles existentes).

+
Documente as decisões

Explique por que escolheu cada valor. Facilita revisões e auditorias.

+
Revise periodicamente

Scores podem mudar com novas informações ou exploits publicados.

5. Erros Comuns

-
Confundir Attack Complexity com dificuldade do exploit

AC refere-se a condições do AMBIENTE, não à habilidade técnica necessária.

-
Ignorar o Scope

Uma vulnerabilidade que permite escapar de um sandbox (S:C) é muito mais grave.

-
Superestimar impactos "possíveis"

Avalie o impacto DIRETO da vulnerabilidade, não cenários hipotéticos encadeados.

-
Usar apenas o Base Score

Temporal e Environmental existem para refinar a priorização ao seu contexto.

-
Copiar scores de CVEs sem análise

O score do NVD é genérico. Seu ambiente pode ter fatores atenuantes ou agravantes.
