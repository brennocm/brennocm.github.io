Definitive Guide: How to Classify Vulnerabilities

1. What is CVSS?

The Common Vulnerability Scoring System (CVSS) is an open and standardized framework for assessing the severity of security vulnerabilities in systems. Developed by FIRST (Forum of Incident Response and Security Teams), CVSS provides a numerical score from 0.0 to 10.0 that reflects the severity of a vulnerability.

Why use CVSS?

  • Standardization: Common language between security, development, and management teams
  • Prioritization: Helps decide which vulnerabilities to fix first
  • Communication: Makes it easier to report risks to non-technical stakeholders
  • Compliance: Required by standards like PCI-DSS, ISO 27001, and GRC frameworks

Severity Table

Score Severity Recommended Action
0.0 None Informational only
0.1 - 3.9 Low Fix in normal cycle
4.0 - 6.9 Medium Priority fix
7.0 - 8.9 High Urgent fix
9.0 - 10.0 Critical Immediate fix

2. Understanding the Metrics

Attack Vector (AV)

Defines how the attacker can reach the vulnerable component:

Network (N) Score: High

Exploitable remotely via Internet. Example: Exposed web API, public server.

Adjacent (A) Score: Medium-High

Requires access to the same network (Wi-Fi, LAN, Bluetooth). Example: ARP Spoofing on corporate network.

Local (L) Score: Medium

Requires local access to the system. Example: Privilege escalation after SSH login.

Physical (P) Score: Low

Requires physical access to the device. Example: Cold boot attack, malicious USB.

Attack Complexity (AC)

Conditions beyond the attacker's control that need to exist:

Low (L) Score: High

Repeatable attack, no special conditions. Example: Simple reflected XSS.

High (H) Score: Low

Requires specific conditions (timing, special config). Example: Race condition.

Privileges Required (PR)

Level of authentication/authorization before the attack:

None (N) Score: High

No authentication. Example: SQL Injection on login page.

Low (L) Score: Medium

Authenticated common user. Example: IDOR accessing other users' data.

High (H) Score: Low

Administrative privileges. Example: RCE via admin panel.

User Interaction (UI)

Does the attack require action from another user?

None (N) Score: High

Fully automated. Example: Worm, remote exploit.

Required (R) Score: Medium

Victim needs to click/open something. Example: Phishing, persistent XSS + click.

Scope (S)

Does the vulnerability affect other components beyond the vulnerable one?

Unchanged (U)

Impact limited to component. Example: DoS on a specific service.

Changed (C)

Affects other systems/components. Example: VM escape, XSS stealing cookies from another domain.

Impacts (CIA Triad)

Evaluate each dimension of the CIA triad:

  • Confidentiality (C): Unauthorized access to information
  • Integrity (I): Unauthorized modification of data
  • Availability (A): Service or system interruption

For each: None (N) = no impact | Low (L) = partial impact | High (H) = full impact

4. Best Practices

+
Be consistent

Use the same criteria across the organization. Create an internal classification guide.

+
Consider the context

Use Environmental metrics to adjust to your environment (asset criticality, existing controls).

+
Document decisions

Explain why you chose each value. Makes reviews and audits easier.

+
Review periodically

Scores can change with new information or published exploits.

5. Common Mistakes

-
Confusing Attack Complexity with exploit difficulty

AC refers to ENVIRONMENT conditions, not the required technical skill.

-
Ignoring the Scope

A vulnerability that allows escaping a sandbox (S:C) is much more severe.

-
Overestimating "possible" impacts

Evaluate the DIRECT impact of the vulnerability, not hypothetical chained scenarios.

-
Using only the Base Score

Temporal and Environmental exist to refine prioritization to your context.

-
Copying CVE scores without analysis

The NVD score is generic. Your environment may have mitigating or aggravating factors.
